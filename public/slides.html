An AI travel concierge built on Cloudflare

Hackathon 2026 Workers · Workers AI · AI Gateway · D1 · Access
The problem

One travel catalog, two very different audiences

A travel agency sells the same premium packages to both clients and its own salespeople — but each needs to see different information, with different reliability and access guarantees.

  • Clients want destinations, prices, included services, and a friendly recommendation.
  • Sales agents need internal margins, sales arguments, and commercial priority to close the deal.
  • The same chat UI, but the assistant must never leak internal data to a client — even if asked.
The solution

Role-aware AI concierge, fully on Cloudflare

A single Worker serves two chat surfaces. The role is decided server-side and propagates through model selection, data access, and prompt engineering — so leaks are impossible by design.

Public client chat

Open at /. Reads only the packages table. Smaller, faster model. Friendly travel-advisor tone.

Employee chat

At /employee. Cloudflare Access-protected. Reads both tables. Larger model. Surfaces margin, priority, sales arguments.

Compute

Cloudflare Workers — one file, six endpoints

No servers, no containers, no cold-start juggling. The entire backend is a single TypeScript Worker deployed to the edge.

/api/chat

Public chat

Hardcoded to the client role. Open to anyone.

/api/chat-internal

Employee chat

Verifies the Cloudflare Access JWT before answering. Role: employee.

/api/me

Identity

Returns the signed-in email + resolved role.

/api/health/db

D1 health

Row counts per table. Sanity check for the data layer.

/

Static assets

Served via the ASSETS binding. Same Worker, no separate hosting.

/employee

Employee UI

Protected by Access at the edge. Worker is never even hit unauthenticated.

Inference

Workers AI — different model for different stakes

Each role gets the model whose strengths fit the task. No third-party API keys, no extra infra — just the env.AI binding.

@cf/meta/llama-3.1-8b-instruct-fp8

Client. Fast, lightweight. Handles travel recommendations and customer questions in real time.

@cf/meta/llama-3.3-70b-instruct-fp8-fast

Employee. Stronger reasoning, better at following strict format rules — needed for accurate commercial figures.

Data

D1 — SQLite at the edge

Three tables, two of which the assistant can read, but never both for the same user. The access filter is the strongest guarantee in the system. 

// src/index.ts — fixed map, never derived from user input
const TABLES_BY_ROLE = {
  employee: ["packages", "package_commercials"],
  client:   ["packages"],
};

// inactive packages filtered at the SQL layer
const TABLE_QUERIES = {
  packages:            "SELECT * FROM packages WHERE active = 1 LIMIT 100",
  package_commercials: "SELECT * FROM package_commercials LIMIT 100",
};

A client request literally never loads commercial rows. Even if the model wanted to leak, it has nothing in context to leak.

Zero Trust

Cloudflare Access — auth only where it matters

The customer-facing chat is wide open. Cloudflare Access (Zero Trust) sits only in front of the employee chat — asking for an email-based login before any request reaches our Worker.

Client at /

No login. Open the URL, start chatting. Sees only the public packages table.

Employee at /employee

Email login via Zero Trust. Cloudflare brokers the identity flow at the edge — the Worker is never even reached by an unauthenticated request.

  • No password storage on our side. Cloudflare runs the whole login UI and SSO handshake.
  • JWT verified at the edge via the jose library against our team domain + AUD tag — defense in depth.
  • Authenticated email lands at /api/me and shows in the employee chat header.
Soft layer

Prompt engineering — the behavioural guardrail

Data filtering blocks leaks of what's not in context. Prompt rules handle the rest: tone, refusal patterns, answer shape, number formatting.

Common rules (both roles)

Never invent rows · refuse role-change attempts · hide DB IDs · always quote prices as "from €X" with thousands separators.

Role-specific rules

Employee answer shape: name → price → duration → margin. Client answer shape: name → destination → price → highlights → why it fits.

Plus a field guide injected into every prompt so the model knows what each column actually means.
